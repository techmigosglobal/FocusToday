## 1. Project Overview

CRII is an offline-first, Firebase-backed social/news platform for India, built with Flutter. It features:
- Phone-based OTP authentication via Msg91
- A vertical flip-page news feed (Way2News style)
- Role-based content creation and moderation workflow
- Firebase Cloud Functions backend (Node.js 22, deployed `asia-south1`)
- Multi-device FCM push notification system

### Architecture Summary

```
Flutter (Dart)
  ├── lib/features/auth          ← OTP auth, splash, onboarding
  ├── lib/features/feed          ← News feed, post CRUD, moderation
  ├── lib/features/moderation    ← Admin approve/reject dashboard
  ├── lib/features/notifications ← FCM inbox, preferences
  ├── lib/features/profile       ← User profile, bookmarks
  ├── lib/features/workspace     ← Reporter workspace, campaigns
  ├── lib/core/services          ← Notification, Msg91, Firestore, Cache
  └── lib/shared/models          ← Post, User, Comment, Meeting

Firebase Backend (TypeScript → Node.js 22, asia-south1)
  ├── firebase_backend/functions/src/index.ts             ← Core callable functions
  ├── firebase_backend/functions/src/notification_triggers.ts  ← Firestore triggers + FCM
  ├── firebase_backend/firestore/                         ← Firestore rules + indexes
  └── firebase_backend/storage/                           ← Storage rules
```

## 2. OTP Authentication — Full Analysis

### Flow

```
User enters 10-digit phone
        │
        ▼
PhoneLoginScreen → Msg91Service.sendOTP("91<phone>")
        │           (Uses sendotp_flutter_sdk OTPWidget)
        ▼
Returns { request_id: "..." }
        │
        ▼
Navigate to OTPVerificationScreen (reqId passed)
        │
User types 6-digit OTP (auto-focus advance, auto-verify on 6th digit)
        │
        ▼
Msg91Service.verifyOTP(reqId, otp)
        │
Returns { type: "success", message/accessToken: "<JWT>" }
        │
        ▼
AuthRepository.verifyAndSaveSession(phone, accessToken)
        │
        ▼ calls FirebaseFunctions.instanceFor(region: 'asia-south1')
           .httpsCallable('verifyMsg91OtpAndExchangeToken')
        │
        ▼
Cloud Function (index.ts):
  1. verifyMsg91AccessToken() — tries Msg91 API, falls back to JWT decode
  2. findUserByPhone() — checks phone_normalized / phone_number in Firestore
  3. Creates/updates user doc in Firestore
  4. ensureFirebaseUser() — creates Firebase Auth user
  5. setCustomUserClaims({ role, phone_number })
  6. createCustomToken(userId)  → returns to client
        │
        ▼
Flutter: FirebaseAuth.signInWithCustomToken(customToken)
        │
        ▼
Session persisted in SharedPreferences + SecureStorage
NotificationService.onUserAuthenticated(userId, role)
  → syncFcmTokenToServer → subscribeToRoleTopics
        │
        ▼
Navigate to MainNavigationShell
```

## 3. News Feed — Full Analysis

### Architecture

The feed uses a custom vertical flip-page view (`FlipPageView`) with gesture-driven drag animations, mimicking the Way2News experience.

**Data flow:**
```
PostRepository.getApprovedPostsWithInteractions(userId)
  → Firestore query: posts WHERE status == 'approved' ORDER BY published_at DESC
  → Firestore: posts/{id}/interactions/{userId} for like/bookmark state
  → Returns List<Post> with isLikedByMe, isBookmarkedByMe populated

FeedScreen loads posts → renders in FlipPageView
  → Vertical drag: DragUpdate adjusts _dragProgress (0.0 to 1.0)
  → DragEnd: velocity-based decision to flip or snap back
  → After flip: _currentIndex++ / -- and reset progress

Each card → VerticalContentCard
  → Shows: caption, media, author, category, timestamp
  → Actions: like, comment, bookmark, share
  → All interactions go through Cloud Functions (togglePostInteraction)
  → Optimistic UI with server rollback on failure
```

### Content Types Supported
- **Image** — CachedNetworkImage
- **Video** — VideoPlayer with FeedVideoControllerService
- **Article** — ArticleReaderScreen with full text
- **PDF** — SyncFusion PDFViewer
- **Poem** — Verse-based render

### Post Interactions
All interactions (like/bookmark) go through `togglePostInteraction` Cloud Function (Firestore transaction) ensuring atomic counter updates. Share goes through `trackShareInteraction`. Comments through `createComment`. Impressions tracked locally with dedup.

## 4. Approval & Rejection Workflow — Full Analysis

### Roles & Permissions

```
public_user (weight: 1)  — Can read feed, like, comment, bookmark
reporter    (weight: 2)  — Can create posts (submitted as "pending")
admin       (weight: 3)  — Can moderate posts + create posts (auto-approved)
super_admin (weight: 4)  — Full access, can assign admin role
```

### Post Lifecycle

```
Reporter creates post
        │
        ▼ Cloud Function: createPost
  status = "pending" (for reporter)
  status = "approved" (for admin/super_admin)
        │
        ▼ Firestore trigger: onPostCreated
  Notifies all admin FCM tokens
  Payload: { type: "new_post_pending", post_id: "..." }
        │
Admin opens ModerationScreen → Pending tab
        │
Admin taps Approve OR Reject
        │
        ▼ PostRepository.updatePostStatus
           → Cloud Function: moderatePost
  Sets: status, rejection_reason, moderated_by, moderated_at
        │
        ▼ Firestore trigger: onPostStatusChange
  Notifies post AUTHOR FCM tokens
  approved: { type: "post_approved", post_id: "..." }
  rejected: { type: "post_rejected", post_id: "...", reason: "..." }
        │
If rejected → reporter sees post in "Rejected Posts" screen
        │
Reporter edits via EditResubmitScreen
        │
        ▼ Post status set back to "pending"
        ▼ Firestore trigger: onPostResubmitted
  Notifies admins again
```

### Issues Found

**[MOD-01] — MEDIUM: `onPostResubmitted` trigger conflicts with `onPostCreated`**  
Both `onPostCreated` (uses `onDocumentCreated`) and `onPostResubmitted` (uses `onDocumentWritten`) can both fire when a resubmitted post goes from `rejected` → `pending`. The `onPostCreated` only fires on initial creation so this is technically fine, but worth verifying. The `onPostResubmitted` correctly checks `beforeStatus === 'rejected' && afterStatus === 'pending'` so it won't double-fire.

**[MOD-02] — LOW: No audit trail display in ModerationScreen**  
`audit_logs` collection exists in Firestore but is not surfaced in the ModerationScreen. The `AuditTimelineScreen` exists but is behind the analytics section — consider linking from the moderation card.

### Flutter-Side Notification Service

- Requests `alert`, `badge`, `sound` permissions on init
- FCM token obtained and stored in `SharedPreferences`
- Token refreshed via `_messaging.onTokenRefresh` listener
- On auth: token synced to `users/{uid}/devices/{deviceId}` + legacy `users/{uid}.fcm_token`
- Role-based FCM topics: `role_admin`, `role_superAdmin`, `role_reporter`, `role_publicUser`, `new_content`
- On logout: device subcollection doc deleted, token deleted, topics unsubscribed
- Navigation from push: `consumePendingNavigation()` checked in `FeedScreen.initState`

### Issues Found

**[NOTIF-01] — CRITICAL: Double `admin.initializeApp()` in Functions**  
`notification_triggers.ts` (line 11) calls `admin.initializeApp()` AND `index.ts` (line 4) also calls `admin.initializeApp()`. Since `index.ts` exports all notification trigger functions, both modules run in the same Node process, causing `"Firebase App named '[DEFAULT]' already exists"` on cold start. **Fixed in Section 6.**

**[NOTIF-02] — HIGH: No foreground message handler in Flutter**  
`notification_service.dart` handles token management and background messages via `firebaseMessagingBackgroundHandler`, but there is **no `FirebaseMessaging.onMessage.listen()` handler** for foreground messages. When the app is in the foreground and a notification arrives, it will be silently dropped — the user won't see it.  
**Fixed in Section 6.**

**[NOTIF-03] — MEDIUM: `notification_service.dart` uses hardcoded `'current_user_id'` key**  
The token refresh handler reads `prefs.getString('current_user_id')` but the auth session stores the user ID under `'user_id'`. This means token refresh after a background token rotation won't re-sync to the server.  
**Fixed in Section 6.**

**[NOTIF-04] — MEDIUM: WhatsApp `integrated_number` is a placeholder**  
`msg91_service.dart` line 163 has `'integrated_number': '919XXXXXXXXX'`. This placeholder will cause all WhatsApp notifications to fail silently.  
**Action Required:** Replace with your actual Msg91 WhatsApp integrated number.

**[NOTIF-05] — LOW: `getAllActiveTokens()` iterates all users**  
For broadcast campaigns, the function fetches ALL user docs and then ALL device subcollections one-by-one. At scale (>1000 users), this will hit Firestore read limits and be very slow. Consider using FCM topics (`new_content`) for broadcasts instead of iterating tokens.

**[NOTIF-06] — LOW: `getRoleTokens()` has N+1 subcollection reads**  
For each user in a role, it fires a separate devices subcollection query. Should be batched or replaced with a Firestore collection group query.
